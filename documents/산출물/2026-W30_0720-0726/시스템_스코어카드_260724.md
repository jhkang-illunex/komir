## 0. 시스템 구조 (2단 분할)

사용자 확정(2026-07-24): 시스템은 **2개로 분할**되며, 5개 파이프라인 단계는 이 2개
시스템에 다음과 같이 속한다.

```
시스템 ① 수집기 시스템
  └─ (정형) 관세청·ECOS·KOMIS 수집기 + (비정형) GDELT GKG 벌크 수집기
     ※ 차후 보안 모듈 추가 예정

시스템 ② 분석 시스템
  ├─ 2-1. 데이터 분석·전처리(GKG 관련성 필터링 등)
  │      ※ 수집기의 보안 모듈이 이 모듈에도 적용될 가능성 있음 — 염두에 둘 것
  ├─ 2-2. 지정학위기지수 생성기
  ├─ 2-3. 5종 광물 수급위기 진단기
  └─ 2-4. 5종 광물 1년후 수요·가격 예측기
```

**보안 모듈 관련 메모**: 아직 설계·구현 전이라 이 스코어카드에는 반영하지 않았다.
수집기 시스템에 먼저 추가될 예정이고, 분석 시스템의 2-1(데이터 분석·전처리)에도
동일 모듈이 적용될 가능성이 있다고 전달받았다 — 향후 버전에서 보안 모듈이 추가되면
각 모듈 항목에 "보안" 지표 행을 신설한다.

## 1. 숫자 읽는 법 — 단계마다 "건수"의 단위가 다르다 ⚠ 필독

이 문서에는 361,407 / 1,815,184 / 295,157처럼 자릿수가 비슷한 숫자가 여러 번
나오는데, **서로 다른 것을 세는 숫자**라 혼동하기 쉽다. 처음 보는 사람이 "왜 문서마다
숫자가 다르지?"라고 의문을 갖지 않도록 아래 변환 흐름을 먼저 명시한다(GDELT GKG
경로 기준, 2026-07-24 실측):

```
① 원본 파일        361,407개 zip 파일 (2.4TB, 2016~2026년, 15분 단위 1개씩)
                    ← 수집기 시스템의 산출물. "이벤트"가 아니라 "파일" 개수.
                       zip 1개 안에는 전 세계 모든 주제의 뉴스 레코드가
                       수천 건씩 들어있어 이벤트 건수와 직접 비교 불가.
   │ 파싱(zip을 열어 CU/NI는 GDELT 테마코드, CO/LI/REE는 키워드로 후보 추출)
   ▼
② 원본 후보 이벤트  1,815,184건 (파싱은 됐지만 아직 정제 전 — 관련성 28.6%,
                    즉 100건 중 71건은 실제로 무관한 기사가 섞여 있는 상태)
   │ 규칙기반 필터 4라운드 + LLM 재검증 2단계 (분석 시스템 2-1 모듈)
   ▼
③ 최종 이벤트       295,157건 (2-1 전처리 모듈의 최종 산출물, 관련성 99.5%)
                    ← 이후 모든 문서·이 표에서 "이벤트 건수"라고 하면 이 숫자.
                       "29.5만"·"295,157"이 바로 이것이다.
```

**정리하면**: "36만 1천"은 ①**수집기가 받아온 원본 파일 개수**, "29만 5천"은
③**전처리를 다 마친 뒤 실제로 쓰는 이벤트 건수**다. 같은 파이프라인의 서로 다른
두 단계를 각각 정확하게 나타낸 것이라 모순이 아니다 — 다만 자릿수가 비슷해서
헷갈리기 쉬우므로, 아래 표부터는 숫자 옆에 **무엇을 세는 숫자인지 괄호로 항상
명시**한다.

(산출근거: 수급위기지수 99/100(model)) 확률: 심각 62%, 주의 22%. 기여: y_lag1 +33.1, price_z52 +3.2, import_hhi -0.6. 관련 이벤트: 'First Quantum has cleared the way...'(Panama, sev 3.0/3)" — 등급 정의·산출근거 점수·단계별 확률분포·상위 기여 피처(부호 포함)·연관 지정학 이벤트 인용까지 한 문장에 전부 담김. ③ `evidence_json` 컬럼에 구조화 버전(전체 `stage_probs`·`contrib`·`override_applied` 등)도 동시 저장돼 있어 사람이 읽는 설명(reason)과 기계가 읽는 근거(evidence_json)를 둘 다 갖춤. **"설명가능한 컨텐츠 생성"이 이미 실제로 되고 있는 상태** — 발주처 보고서·대시보드에 그대로 인용 가능한 수준. |
